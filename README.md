# Cloud Task Manager

A full-stack, serverless task manager built on AWS — started as a beginner learning project and evolved feature by feature into a genuinely complete daily-use application.

## Live site
https://aman-taskmanager-2026.s3-website.ap-south-1.amazonaws.com

## Features

- **Authentication** — sign up, email verification, sign in, and persistent sessions (Amazon Cognito)
- **Full CRUD tasks** — create, read, update, delete, scoped so each user only ever sees their own data
- **Due dates** — optional due date per task, with a clear overdue indicator once the date has passed
- **File attachments** — attach a file to any task, uploaded securely and directly to S3 using presigned URLs
- **Automated email reminders** — a daily scheduled check emails a summary of tasks due that day
- **Recurring tasks** — mark a task as daily/weekly; completing it resets it and pushes the due date forward instead of deleting it
- **Clear completed** — bulk-remove every checked-off task in one click
- **Focus mode** — a toggle that hides everything except what's due today, for days when the full list feels like too much

## Architecture

| Service | Role |
|---|---|
| **Amazon S3** | Hosts the static frontend; separate private bucket stores uploaded files |
| **AWS IAM** | Secure, least-privilege access for both the developer account and each Lambda function |
| **Amazon Cognito** | User sign-up, sign-in, and JWT-based session tokens |
| **Amazon API Gateway (HTTP API)** | Public REST-style endpoints, protected by a Cognito JWT authorizer |
| **AWS Lambda** | All backend logic — one function per action (see `/lambda`) |
| **Amazon DynamoDB** | Stores tasks, with per-user ownership enforced at the database level via `ConditionExpression` |
| **Amazon SNS** | Sends email notifications |
| **Amazon EventBridge Scheduler** | Triggers the daily due-task check automatically |

## How it fits together

1. The browser loads the frontend from S3.
2. A user signs up/signs in through Cognito, receiving a JWT.
3. Every API call includes that JWT in the `Authorization` header.
4. API Gateway verifies the JWT before ever reaching Lambda — unauthenticated requests are rejected outright.
5. Lambda functions read the verified user ID from the token (never trusting a client-supplied ID) and read/write only that user's data in DynamoDB.
6. File uploads go **directly from the browser to S3** using a short-lived presigned URL generated by Lambda — files never pass through Lambda itself.
7. A scheduled EventBridge rule invokes a Lambda function once a day, which scans for tasks due that day and publishes a summary to an SNS topic, which emails the user.
